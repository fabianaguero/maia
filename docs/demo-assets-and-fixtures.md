# Demo Assets and Fixtures

This repository contains product code, but it also includes demo-oriented assets used for prototyping, screenshots, auditory experiments, and analyzer verification.

Use this file to understand what these artifacts are before deleting or publishing them.

## Demo Audio Renders

Examples:

- `demo/audio/advanced_glitch_test.mp3`
- `demo/audio/ai_glitch_test.mp3`
- `demo/audio/heartbeat_demo.mp3`
- `demo/audio/java_anyma_remix.mp3`
- `demo/audio/java_crash_sonification.mp3`
- `demo/audio/normal_deploy_sonification.mp3`

These are rendered demo outputs or verification artifacts, not source code.
They are useful for product storytelling and quick auditory validation, but they should be reviewed for redistribution intent before public release.

Current repo note:

- these MP3s are not wired into the desktop runtime as product assets
- they are closer to demo renders and verification outputs than to shipping application resources

Related generator scripts:

- `scripts/verification/generate_heartbeat_demo.py`
- `scripts/verification/verify_ai_glitch.py`
- `scripts/verification/verify_hybrid_glitch.py`
- `scripts/verification/verify_hybrid_remix.py`
- `scripts/verification/verify_normal_log.py`
- `scripts/verification/verify_real_crash.py`

## Log Fixtures

Examples:

- `analyzer/fixtures/logs/sample-production.log`
- `analyzer/fixtures/logs/*`
- `desktop/test/logs/*`

These exist to simulate operational signal for analysis, tests, and demos.
They should be treated as fixtures and should remain synthetic or anonymized.

## Generated Local Helper Output

Example:

- `.run/demo/maia_live.log`
- `.run/verification/*`

`.run/demo/maia_live.log` is generated by `scripts/demo/tropical_log_gen.py`.
`.run/verification/*` is generated by the helpers under `scripts/verification/`.
These are local helper artifacts, not durable fixtures.

## Guidance

If you are preparing the repository for public release:

1. Keep assets that help explain Maia and verify the product.
2. Remove unrelated local artifacts and machine-specific files.
3. Confirm that any retained media is intentionally redistributable.
4. Prefer documenting fixture intent rather than leaving ambiguous root-level binaries unexplained.
